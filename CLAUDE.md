# CLAUDE.md — 주마등 (Chrono Zoetrope)

이 파일은 Claude Code가 매 세션 시작 시 읽는 프로젝트 컨텍스트다. 아래 확정 사항과 원칙을 기준선으로 삼는다. 미결 사항은 명시된 대로 열어둔다.

---

## 0. 이 프로젝트가 무엇인가 (한 문단)

사용자 자신의 생애를 주마등처럼 빠르게 흐르는 몽타주로 제시하고, 사용자가 원하는 순간에 버튼을 눌러 멈춰 그 순간의 1인칭 view로 진입하는 1인용 몰입 설치다. "내 삶이 스쳐 지나가고 그중 한 순간에 다시 들어가 본다"는 임사체험 은유를 골격으로 한다. 지름 140cm 반투명 실린더 안에 사용자가 서고, 프로젝터 네 대가 바깥에서 후면 투사한다.

---

## 1. 절대 원칙 (해석적 자율성)

이건 협상 불가능한 연구 기준선이다. 코드 결정이 이 원칙과 부딪히면 원칙이 이긴다.

- 시스템은 빈 해석 틀을 주고 침묵한다. 미래나 과거의 의미를 대신 말하지 않는다.
- 주마등 몽타주는 너무 빨라 온전히 읽히지 않는다. 의미를 잡으려면 사용자가 스스로 멈춰야 한다. 이 "읽히지 않음"은 결함이 아니라 설계다.
- 순간 진입 view에는 내레이션, 자막, 설명, 음성 해설을 절대 붙이지 않는다. 시선만 있다.
- direct delivery(서사를 완성해 떠먹이는 것)를 피한다. 진입 순간에 텍스트나 해설을 붙이는 설계 제안은 거부한다.
- AI로 "이 사람의 생애를 어떤 장면으로 구성할지"를 판단하게 하는 자리가 생기면, 서사를 완성하는 역할이 아니라 해석 가능한 재료를 흩어놓는 역할로 제한한다.

## 2. 물리 구조 및 광학 (확정)

- 구조물: 지름 140cm(반경 70cm), 높이 140cm 반투명 실린더.
- 설치 고도: 바닥에서 +100cm 띄움. 사용자는 고소작업차(리프트)로 올라와 안에 선다.
- 프로젝터: 4대, 실린더 바깥에 사방 배치, 막에 **후면 투사**.
- 담당 영역: 각 프로젝터가 둘레의 90도 호를 맡는다. 인접 둘이 45도 대각 네 지점에서 겹쳐 블렌딩한다.
- 반투명 막이 디퓨저로 작동해 블렌딩과 초점 차를 부드럽게 확산시킨다.
- 프로젝터 해상도: FHD 1920×1080.
- 세로 프레이밍: 실린더 세로 전체(140cm)를 영상으로 채운다. 바닥과 천장(설치물 밖)은 어둡게 둔다.

## 3. 개발 스택 (확정)

웹 스택으로 통일한다. Unreal / nDisplay는 채택하지 않는다. 실시간 3D 물리나 형태 변형 액추에이터가 없고, 하는 일은 파노라마를 실린더에 재투영하고 컨트롤러 입력으로 상태를 전환하는 것이라 웹이 더 단순하다. (2026-07 Electron → 순수 Node.js 웹앱으로 전환. Electron이 하던 멀티 디스플레이 자동 풀스크린은 단일 창 4타일로 대체됐다.)

- 런타임 셸: **Node.js HTTP 서버 + 브라우저**. 서버(`server/index.mjs`)가 상태 기계·미디어·SSE를 소유하고, 브라우저 페이지 한 개가 4개 프로젝터 뷰를 한 캔버스에 2×2 타일로 렌더한다.
- 백엔드 로직: **Node.js** (상태 기계, ComfyUI 클라이언트, 상태 허브). main↔renderer IPC는 SSE(`/api/events`, server→page) + HTTP POST(page→server)로 대체.
- 미디어 서빙: 과거 `zoe://` Electron 프로토콜 → 서버 동일 오리진 **`GET /media/*`** (CORS·Range).
- 렌더러: **Three.js** (WebGL, 실린더 재투영, 후면 투사 반전, 블렌딩 셰이더). 빌드는 **Vite**(`npm run build` → `dist/`, 서버가 정적 서빙; dev는 `npm run dev` = 서버 + vite HMR 프록시).
- 입력: **Gamepad API** (Xbox Wireless Controller, USB-C 유선) + 페이지 keyboard(Enter/V/Space). 단일 창이라 Electron globalShortcut 불필요.
- AI 서버: 로컬 **ComfyUI** (`http://143.248.107.38:8188/`)
- 이미지 생성: **SDXL + ControlNet(depth)** 파노라마

## 3.5 프로젝션 출력 시스템 (런타임 하드웨어 체인)

개발과 실행 머신이 다르다. 개발은 macOS(§11), **설치 구동은 Windows 노트북**에서 한다.
아래 MST·GPU 출력 한계·USB-C DP 동작은 전부 이 Windows 런타임 머신 고유 특성이다.
디스플레이 출력 코드(후면 투사 반전, 4타일 배치·프로젝터 배정)는
반드시 이 런타임 머신에서 검증한다. Mac에서의 동작은 참고용일 뿐이다.

**웹앱 전환 후 운용**: 단일 브라우저 창(권장: Chrome `--kiosk`)을 열어 `http://<서버>:8788` 에 접속하면
한 화면에 4타일(P0|P1 위, P2|P3 아래)이 렌더된다. 각 프로젝터에는 이 창의 해당 사분면이 나가도록

설정(확장/미러)으로 잡는다.

## 4. 렌더링 파이프라인

### 4.1 실린더 재투영 모델
씬 안에 실제 치수(반경 70cm, 높이 140cm)의 실린더 메시를 세운다. 파노라마를 실린더 안쪽 면에 원통 UV로 감는다. 파노라마 가로축(경도)이 실린더 둘레(방위각)에 1:1 선형 대응하므로 수평 왜곡이 구조적으로 없다.

네 대의 실제 프로젝터 위치에 가상 카메라를 각각 둔다. 각 가상 카메라는 담당 프로젝터의 렌즈 스펙(스로우비, 렌즈 오프셋)대로 실린더를 렌더한다. 이 렌더 결과가 그 프로젝터가 쏴야 할 사전 왜곡 영상이다. 곡률 보정, 키스톤, 오프액시스가 이 한 번의 렌더에 자동으로 들어간다.

뷰어 위치는 렌더에 개입하지 않는다. 표면에 상을 칠하는 방식이라 헤드트래킹이 필요 없다. 캘리브레이션은 가상 카메라 포즈를 실제 프로젝터 포즈에 맞추는 작업이다.

### 4.2 후면 투사 반전
프로젝터가 막 뒤에서 쏘므로 상이 거울상이 된다. 각 창 최종 출력에 수평 플립을 적용한다. 프로젝션 행렬에서 x축 반전 또는 포스트 셰이더에서 UV.x 반전. 네 대 반전을 소프트웨어에서 일관되게 관리하기 위해 셰이더 단 처리를 기본으로 한다.

### 4.3 엣지 블렌딩
인접 프로젝터 담당 호를 약간 겹치게 배치하고, 겹침 구간을 셰이더 알파 그라디언트로 페더링한다. 후면 투사 막 확산이 블렌딩을 관대하게 만든다. 정밀 물리 정렬이 필요하면 Three.js 출력을 MadMapper로 다운스트림 워핑하는 경로를 안전판으로 둔다.

### 4.4 해상도 전략
FHD 한 대가 110cm 호를 담당하면 약 17.5px/cm. 소스 파노라마를 8192×4096으로 뽑으면 한 사분면이 약 2048px를 제공해 업스케일 손실을 줄인다.

## 5. AI 파이프라인 (두 갈래)

시간 예산이 정반대라 이원화한다.

### 5.1 사전 생성 라이브러리 (주마등 몽타주용)
주마등은 빠르게 흘러야 한다. 실시간 생성(8~15초/장)으로는 불가능하다. 세션 초반(인트로 또는 상승 시퀀스)에 생애 몽타주용 파노라마를 미리 생성해 캐시한다. 주마등 모드는 이 캐시를 빠르게 재생하는 텍스처 교체 루프다.

### 5.2 실시간 재생성 (순간 진입용)
멈춤 시점에 그 순간을 1인칭 view로 ComfyUI에 실시간 재생성 요청한다. 8~15초 지연은 결함이 아니라 자산이다. 생성 대기가 해석적 기다림의 의례로 기능한다. (리프트 연출 확정 시 이 대기를 몸의 하강으로 신체화하는 옵션과 결합. 9절 참조.)

### 5.3 ComfyUI 통신 (MCP 아님, 직접 HTTP/WS)
런타임 코어는 MCP를 쓰지 않는다. 에이전트적 도구 선택이 없는 결정론적 파이프라인이라 직접 프로토콜이 더 단순하고 빠르다.

- 워크플로우 그래프를 `/prompt`로 POST → `prompt_id` 수신
- `/ws`로 실행 진행률·완료 이벤트 수신
- `/history/{prompt_id}`로 출력 노드 결과 조회
- `/view`로 이미지 파일 로드 후 텍스처화

## 6. 상태 기계

Node 메인 프로세스가 단일 상태 기계를 돌린다. 각 전이는 IPC로 네 렌더러 창에 브로드캐스트한다.

1. **IDLE** — 대기. 다음 사용자를 기다린다.
2. **ENTRY** — 사용자 진입, 리프트 상승. 이 구간에서 주마등 라이브러리 사전 생성 시작.
3. **ZOETROPE** — 주마등 몽타주 재생. 생애가 빠르게 흐른다. 멈춤 입력 대기.
4. **FREEZE** — 멈춤 입력 수신. 현재 순간 포착, 실시간 재생성 요청.
5. **REGEN_WAIT** — AI 재생성 대기. 해석적 기다림의 의례. (리프트 하강 신체화 결합 지점.)
6. **IMMERSION** — 1인칭 view 표시. 자막·해설 없음. 재개 또는 진입 반복 대기.
7. **EXIT** — 퇴장, 리프트 하강. 종료 시퀀스 후 IDLE 복귀.

---

## 7. 오케스트레이션과 창 동기화

메인 프로세스가 상태와 콘텐츠 포인터(현재 파노라마 ID, 몽타주 프레임 인덱스, 멈춤/재생 플래그)를 소유한다. 상태 변화는 IPC로 네 창에 동시 전파. 각 창은 자기 담당 가상 카메라 하나만 렌더한다.

주마등의 빠른 텍스처 교체는 창 간 수 ms 오차에 관대하다. 다만 FREEZE 시점 텍스처 스왑은 네 창이 같은 프레임을 잡아야 하므로, 이 순간만 명시적 배리어(전 창 준비 완료 확인 후 동시 커밋)를 둔다.

## 8. 입력

Xbox Wireless Controller를 Gamepad API로 받는다. USB-C 유선 연결로 지연·끊김을 줄인다.

- 멈춤/진입 버튼 1개 (ZOETROPE → FREEZE)
- 재개/퇴장 버튼 1개 (IMMERSION → ZOETROPE 또는 EXIT)

DualSense는 어댑티브 트리거 여지가 있어 대안으로 열어둔다.

## 9. 미결 사항 (아직 결정하지 말 것)

**리프트 수직 이동의 연출 의미가 미정이다.** 이게 두 가지에 영향을 준다.

1. 수평선 정렬: 이동 중 프레이밍이 실제로 바뀌게 둘지, 아니면 리프트 높이를 렌더러가 받아 콘텐츠를 세로로 밀어 수평선을 눈높이에 고정할지. 후자는 리프트 위치 신호를 렌더에 물리는 결합을 추가한다.
2. 상태 기계 REGEN_WAIT: 순간 진입 하강을 채택하면 이 대기를 리프트 하강과 동기화한다.

후보 방향 셋: (a) 수직축을 생애 타임라인으로 매핑, (b) 큰 상승·하강을 입장(탄생)·퇴장(죽음) 북엔드로 사용, (c) 순간 진입 시 하강으로 재생성 대기 신체화. **잠정 추천은 (b)+(c) 조합.** 확정 전까지 리프트 관련 코드는 인터페이스만 열어두고 구현을 하드코딩하지 않는다.

## 10. 권장 빌드 순서

첫 세션에서 다 하지 말고 뼈대부터. 각 단계가 눈으로 확인되면 다음으로.

1. **스캐폴딩**: Electron 창 하나 + Three.js 실린더 메시에 정적 테스트 파노라마 감기. 재투영과 후면 투사 반전을 눈으로 확인.
2. **멀티 윈도우**: 창 4개로 늘리고 가상 카메라 90도씩 배정. 모니터에서 모서리 연결 검증 (실제 프로젝터는 나중).
3. **상태 기계 + IPC**: 컨트롤러 입력으로 IDLE→EXIT 전이. AI는 목업(캐시 이미지)으로 흐름만.
4. **ComfyUI 연동**: 실시간 재생성 + 사전 생성 라이브러리를 실제 서버에 연결. 서버 워크플로우 의존이 커서 마지막.
